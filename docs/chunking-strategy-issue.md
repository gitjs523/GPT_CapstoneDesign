# Chunk 전략 재정의 및 문제 정리

## 1. 문서 작성 목적
이 문서는 현재 문서 처리 파이프라인에서 사용 중인 Chunk 전략의 한계를 정리하고, 향후 어떤 방향으로 Chunk를 재정의해야 하는지 합의하기 위해 작성한다.

본 프로젝트는 단순 강의자료뿐 아니라, 사용자가 공부할 수 있는 다양한 문서(`PDF`, `PPT`, `PPTX`, 요약 자료, 수험 자료, 공시 자료 등)를 업로드하면 이를 바탕으로 요약, 문제 생성, 정답, 해설을 제공하는 방향을 목표로 한다.

이 과정에서 문서를 어떤 단위로 분리할 것인지는 임베딩 품질, 검색 정확도, 문제 생성 품질에 직접적인 영향을 주기 때문에 Chunk 전략은 핵심 설계 요소에 해당한다.

## 2. 왜 Chunk가 필요한가
문서 전체를 하나의 큰 텍스트로 처리하면 다음과 같은 문제가 발생한다.

- 문서 길이가 길어질수록 임베딩 품질이 떨어질 수 있다.
- 여러 개념이 한 번에 섞여 들어가 검색 정확도가 낮아질 수 있다.
- 생성 모델이 필요한 근거만 선택적으로 읽기 어렵다.
- 문제, 정답, 해설 생성 시 불필요한 정보가 함께 들어가 품질이 흔들릴 수 있다.
- 원문 내에서 어떤 부분을 근거로 사용했는지 추적하기 어려워진다.

따라서 문서는 이후 단계에서 다루기 적절한 크기와 의미를 가진 작은 단위로 나누어 처리해야 한다.

## 3. Chunk가 해야 하는 역할
Chunk는 단순히 긴 문서를 잘게 자르는 역할만 수행해서는 안 된다.
Chunk는 다음 역할을 수행할 수 있어야 한다.

- 임베딩 모델이 해당 텍스트의 의미를 비교적 명확하게 벡터화할 수 있어야 한다.
- 검색 단계에서 특정 주제와 관련된 내용만 정확히 찾을 수 있어야 한다.
- 생성 모델이 문제, 정답, 해설을 만들 때 필요한 문맥을 충분히 제공해야 한다.
- 원문 기준으로 어느 위치에서 나온 내용인지 추적할 수 있어야 한다.
- 나중에 요약, 문제 생성, 정답 생성, 해설 생성 등 다양한 작업에 재사용 가능해야 한다.

즉, Chunk는 `검색을 위한 최소 의미 단위`이면서 동시에 `원문 추적이 가능한 데이터 단위`여야 한다.

## 4. 현재 구현 방식
현재 구현된 방식은 파일 형식에 따라 다음 기준으로 텍스트를 나누고 있다.

- PDF: 페이지(`PAGE`) 단위 분리
- PPT/PPTX: 슬라이드(`SLIDE`) 단위 분리
- 추가 옵션: 문단(`PARAGRAPH`) 단위 분리 가능
- 기본 전략: `AUTO` 선택 시 파일 형식에 따라 자동 결정

이 방식은 구현 초기 단계에서 문서를 일정 단위로 나누고 추출 결과를 확인하기에는 유효하다.
또한 페이지나 슬라이드 단위를 그대로 보존하기 때문에 원문 위치를 추적하기 쉽다는 장점이 있다.

## 5. 현재 방식의 문제점
하지만 현재의 페이지/슬라이드 기반 분리는 최종적인 Chunk 전략으로 보기 어렵다.

- 페이지 하나에 여러 개념이 함께 들어갈 수 있다.
- 하나의 개념이 여러 페이지 또는 여러 슬라이드에 걸쳐 설명될 수 있다.
- 물리적 경계 기준 분리는 의미 단위를 보장하지 않는다.
- 임베딩 모델이 하나의 Chunk에서 여러 개념을 동시에 읽게 될 가능성이 높다.
- 생성 모델도 문제를 만들 때 핵심 개념이 아닌 주변 설명까지 함께 읽게 될 수 있다.

즉, 현재 방식은 `물리적 기준에 따른 임시 분리`에는 적합하지만, `의미 기반 검색 및 생성`을 위한 최종 전략으로는 부족하다.

## 6. Chunk 전략 재정의 방향
향후에는 Chunk를 다음 두 종류로 구분하여 사용하는 방향이 필요하다.

### 6.1 물리적 Chunk
물리적 Chunk는 페이지 또는 슬라이드 같은 원문 단위를 기준으로 나눈 Chunk이다.

예시:
- PDF 3페이지
- PPT 5번 슬라이드

역할:
- 원문 위치 추적
- 출처 보존
- 후처리 시 reference metadata 제공

즉, 물리적 Chunk는 `원문 기준의 위치 정보`를 보존하는 역할을 한다.

### 6.2 개념적 Chunk
개념적 Chunk는 제목, 소제목, 번호 체계, 문단 묶음, 슬라이드 제목, bullet 구조 등 문서의 의미 구조를 기준으로 나눈 Chunk이다.

예시:
- 운영체제 개요
- 프로세스와 스레드
- CPU 스케줄링 알고리즘
- 행정법의 기본 원칙

역할:
- 임베딩 모델 입력
- 검색 대상 데이터
- 문제, 정답, 해설 생성의 주 입력

즉, 개념적 Chunk는 `이 Chunk가 무엇을 설명하는지`가 비교적 명확해야 한다.

## 7. 우리가 사용하기로 한 방향
현재 논의 기준으로는 다음 방향을 목표로 한다.

- 현재의 페이지/슬라이드 기반 분리는 유지한다.
  - 단, 이것은 최종 검색용 Chunk라기보다 원문 위치 보존용 데이터로 본다.
- 이후에는 개념적 Chunk를 별도로 생성한다.
  - 제목, 소제목, 번호 체계, 문단 구조 등을 기준으로 section 단위를 만든다.
- 임베딩 및 검색은 개념적 Chunk를 중심으로 수행한다.
- 물리적 Chunk 정보는 메타데이터로 함께 유지한다.
  - 문서명
  - 페이지 범위
  - 슬라이드 범위
  - section 제목
  - 상위 단원명

즉, 정리하면 다음과 같다.

- 물리적 Chunk: 출처 추적용
- 개념적 Chunk: 임베딩, 검색, 문제 생성용

## 8. 개념적 Chunk를 어떻게 나눌 것인가
현재 기준으로는 별도의 청킹 전용 학습 모델을 두기보다, 우선 규칙 기반으로 시작하는 것이 현실적이다.

우선 고려할 수 있는 규칙은 다음과 같다.

- 제목, 소제목, 장/절/항 구조
- 번호 체계 (`1.`, `1.1`, `(1)`, `가.`, `A.` 등)
- 슬라이드 제목
- 문단 묶음
- bullet hierarchy
- 길이 제한 기반 재분할

다만 모든 문서가 완전히 같은 목차 기호를 사용하는 것은 아니므로, 모든 기호를 하나씩 정의하는 방식보다는 `구조를 나타내는 패턴군`을 정의하는 방식이 더 적절하다.

## 9. 향후 구현 방향
향후 Chunk 전략은 아래 순서로 확장하는 것이 적절하다.

1. 현재의 물리적 Chunk 유지
2. 제목/소제목/번호 체계 기반 section 추출
3. section 단위의 개념적 Chunk 생성
4. section이 너무 길 경우 문단 또는 길이 기준으로 추가 분할
5. 물리적 위치 정보를 메타데이터로 연결
6. 개념적 Chunk를 임베딩 입력으로 사용
7. 검색 결과를 기반으로 요약, 문제, 정답, 해설 생성에 활용

## 10. 결론
현재의 페이지/슬라이드 기반 Chunk는 구현 초기 단계에서 텍스트 추출 결과를 분리하고 추적하기 위한 방식으로는 유효하다.
하지만 프로젝트의 최종 목표가 요약, 문제 생성, 정답 및 해설 제공까지 포함하는 이상, 단순 물리적 분리만으로는 충분하지 않다.

따라서 앞으로는 문서의 구조와 의미를 반영한 개념적 Chunk 전략을 별도로 설계해야 하며,
최종적으로는 `물리적 Chunk + 개념적 Chunk`를 함께 사용하는 이중 구조를 목표로 한다.
